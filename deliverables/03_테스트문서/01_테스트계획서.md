# 테스트 계획서

**프로젝트명**: 푸름대학교 학생성장지원센터 CHAMP (SCMS3)
**작성일**: 2025-11-18
**버전**: 1.0
**작성자**: QA팀

---

## 1. 문서 개요

### 1.1 목적
본 문서는 SCMS3 시스템의 테스트 전략, 범위, 방법, 일정 등을 정의하여 체계적인 테스트를 수행하기 위한 가이드를 제공합니다.

### 1.2 범위
- 기능 테스트 (Functional Testing)
- 통합 테스트 (Integration Testing)
- 성능 테스트 (Performance Testing)
- 보안 테스트 (Security Testing)
- 사용성 테스트 (Usability Testing)

---

## 2. 테스트 전략

### 2.1 테스트 레벨

| 레벨 | 설명 | 담당자 | 도구 |
|------|------|--------|------|
| **단위 테스트** | 개별 메서드/클래스 테스트 | 개발자 | JUnit 5, Mockito |
| **통합 테스트** | 모듈 간 통합 테스트 | 개발자 | Spring Boot Test, MockMvc |
| **시스템 테스트** | 전체 시스템 기능 테스트 | QA | Manual Testing, Selenium |
| **인수 테스트** | 사용자 요구사항 충족 확인 | 사용자 + QA | Manual Testing |

### 2.2 테스트 유형

#### 2.2.1 기능 테스트
- **인증 및 권한**: 로그인, 로그아웃, 비밀번호 변경, 권한 검증
- **프로그램 관리**: CRUD, 신청/취소/승인/거부
- **마일리지**: 적립, 차감, 조회, 랭킹
- **상담**: 신청, 승인, 기록 작성
- **포트폴리오**: CRUD, 공유
- **설문**: CRUD, 응답 제출, 통계
- **외부취업 가점**: CRUD, 승인/거부
- **알림**: 생성, 조회, 읽음 처리

#### 2.2.2 비기능 테스트
- **성능 테스트**: 응답 시간, 처리량, 동시 사용자
- **보안 테스트**: SQL Injection, XSS, CSRF, 세션 관리
- **호환성 테스트**: 브라우저 호환성 (Chrome, Firefox, Safari, Edge)
- **사용성 테스트**: UI/UX, 접근성

## 3. 테스트 범위

### 3.1 테스트 대상

| 모듈 | 테스트 항목 | 우선순위 |
|------|------------|----------|
| 인증 | 로그인, 로그아웃, 비밀번호 관리 | 높음 |
| 프로그램 관리 | CRUD, 신청 관리 | 높음 |
| 마일리지 | 적립, 조회, 랭킹 | 중간 |
| 상담 | 신청, 승인, 기록 | 중간 |
| 포트폴리오 | CRUD, 공유 | 낮음 |
| 설문 | CRUD, 응답, 통계 | 중간 |
| 외부취업 가점 | CRUD, 승인 | 낮음 |
| 알림 | 생성, 조회 | 낮음 |

- 외부 라이브러리 (Spring Framework, JPA 등)
- 데이터베이스 (MySQL) 자체 기능

## 4. 테스트 케이스

### 4.1 인증 테스트

| TC-ID | 테스트 케이스 | 입력 | 예상 결과 | 우선순위 |
|-------|-------------|------|----------|----------|
| TC-001 | 정상 로그인 | 유효한 학번/비밀번호 | 로그인 성공, 메인 페이지 이동 | 높음 |
| TC-002 | 잘못된 비밀번호 | 유효한 학번 + 잘못된 비밀번호 | 로그인 실패, 에러 메시지 | 높음 |
| TC-003 | 존재하지 않는 학번 | 존재하지 않는 학번 | 로그인 실패, 에러 메시지 | 높음 |
| TC-004 | 계정 잠금 | 5회 연속 로그인 실패 | 계정 잠금, 관리자 문의 안내 | 중간 |
| TC-005 | 로그아웃 | 로그인 상태에서 로그아웃 | 세션 무효화, 로그인 페이지 이동 | 높음 |
| TC-006 | 비밀번호 변경 | 현재 비밀번호 + 새 비밀번호 | 비밀번호 변경 성공 | 높음 |
| TC-007 | 비밀번호 재설정 | 학번 + 이름 + 생년월일 | 비밀번호 생년월일로 초기화 | 중간 |

### 4.2 프로그램 관리 테스트

| TC-ID | 테스트 케이스 | 입력 | 예상 결과 | 우선순위 |
|-------|-------------|------|----------|----------|
| TC-101 | 프로그램 목록 조회 | 없음 | 모든 프로그램 목록 표시 | 높음 |
| TC-102 | 프로그램 상세 조회 | 프로그램 ID | 프로그램 상세 정보 표시 | 높음 |
| TC-103 | 프로그램 신청 | 로그인 + 프로그램 ID | 신청 성공, 상태: PENDING | 높음 |
| TC-104 | 중복 신청 방지 | 이미 신청한 프로그램 | 에러 메시지: 이미 신청함 | 중간 |
| TC-105 | 정원 초과 신청 방지 | 정원 초과된 프로그램 | 에러 메시지: 정원 초과 | 중간 |
| TC-106 | 신청 취소 | 대기 중인 신청 | 취소 성공, 상태: CANCELLED | 높음 |
| TC-107 | 승인된 신청 취소 방지 | 승인된 신청 | 에러 메시지: 승인된 신청은 취소 불가 | 중간 |
| TC-108 | 프로그램 신청 승인 (관리자) | 대기 중인 신청 | 승인 성공, 알림 발송 | 높음 |
| TC-109 | 프로그램 신청 거부 (관리자) | 대기 중인 신청 + 사유 | 거부 성공, 알림 발송 | 높음 |
| TC-110 | 참여 완료 처리 (관리자) | 승인된 신청 | 완료 성공, 마일리지 지급 | 높음 |

### 4.3 마일리지 테스트

| TC-ID | 테스트 케이스 | 입력 | 예상 결과 | 우선순위 |
|-------|-------------|------|----------|----------|
| TC-201 | 마일리지 조회 | 로그인 | 총 마일리지 표시 | 높음 |
| TC-202 | 마일리지 적립 내역 조회 | 로그인 | 적립 내역 목록 표시 | 중간 |
| TC-203 | 프로그램 완료 시 마일리지 적립 | 프로그램 완료 처리 | 마일리지 자동 적립 | 높음 |
| TC-204 | 마일리지 수동 지급 (관리자) | 학번 + 점수 + 사유 | 마일리지 지급 성공 | 중간 |
| TC-205 | 마일리지 랭킹 조회 | 로그인 | 전체 랭킹 + 내 순위 표시 | 낮음 |

### 4.4 상담 테스트

| TC-ID | 테스트 케이스 | 입력 | 예상 결과 | 우선순위 |
|-------|-------------|------|----------|----------|
| TC-301 | 상담 신청 | 상담 유형 + 희망 일시 + 주제 + 내용 | 신청 성공, 상태: PENDING | 높음 |
| TC-302 | 상담 승인 (상담사) | 대기 중인 신청 + 일시 + 장소 | 승인 성공, 알림 발송 | 높음 |
| TC-303 | 상담 거부 (상담사) | 대기 중인 신청 + 거부 사유 | 거부 성공, 알림 발송 | 중간 |
| TC-304 | 상담 기록 작성 (상담사) | 완료된 상담 + 요약 + 내용 | 기록 저장, 마일리지 지급 | 높음 |
| TC-305 | 학생 피드백 작성 | 완료된 상담 + 만족도 + 피드백 | 피드백 저장 성공 | 중간 |

## 5. 테스트 환경

### 5.1 하드웨어
- **서버**: Ubuntu 20.04 LTS, 4 Core CPU, 8GB RAM
- **클라이언트**: Windows 10, macOS, 8GB RAM

### 5.2 소프트웨어
- **Backend**: Java 17, Spring Boot 3.x, MySQL 8.0

- 테스트 프로그램: 20개
- 테스트 신청: 50건

## 7. 결함 관리

### 7.1 결함 우선순위

| 우선순위 | 설명 | 예시 |
|----------|------|------|
| **Critical** | 시스템 중단, 데이터 손실 | 로그인 불가, DB 오류 |
| **High** | 주요 기능 동작 불가 | 프로그램 신청 불가 |
| **Medium** | 기능 일부 동작 불가 | UI 깨짐, 필터 미작동 |
| **Low** | 사소한 불편 | 오타, 레이아웃 미세 조정 |

### 7.2 결함 추적 도구
- **GitHub Issues**: 결함 등록 및 추적
- **Label**: Bug, Critical, High, Medium, Low

## 8. 성공 기준

### 8.1 테스트 통과 기준
- **단위 테스트**: 코드 커버리지 80% 이상
- **통합 테스트**: 주요 시나리오 100% 통과
- **시스템 테스트**: Critical/High 결함 0건, Medium 결함 5건 이하
- **성능 테스트**: 평균 응답 시간 2초 이하, 동시 사용자 100명 처리

### 8.2 배포 승인 기준
- 모든 Critical/High 결함 수정 완료
- 사용자 인수 테스트 통과
- 성능 테스트 기준 충족
